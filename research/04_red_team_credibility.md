# Red-Team / Credibility Memo

## thesis
VANTAGE becomes weak the moment it implies more autonomy, prediction, or sensor truth than the prototype can prove. The safest path is to frame it as a decision-support console for one replayed maritime deception workflow, with explicit evidence boundaries and visible human approval.

## recommended_scope
- Keep the system claim at "early warning plus evidence fusion for one operator workflow."
- Treat the four agents as named reasoning stages unless a real runtime exists.
- Make every major evidence item clickable and bounded.
- Use the approval gate as the final credibility anchor.

## top_risks
- "ChatGPT for generals" reaction if the pitch leans on LLMs instead of receipts.
- Prediction skepticism if the Forecaster is treated like a proven model rather than a prototype detector.
- Demo theater exposure if live integrations are mentioned but not visible.
- Credibility loss if the Synthesizer invents motive or intent from sparse evidence.

## top_cuts
- "real-time picture of what adversaries are actually doing" as a V1 claim
- "multi-agent platform" language if the runtime is staged
- any promise of reliable cross-domain intent inference
- any comparison that implies parity with Palantir, Maven, or Lattice breadth

## stage_safe_claims
- "This prototype demonstrates a replayed analyst workflow using a real UI and precomputed evidence."
- "The contradiction logic is heuristic and evidence-backed, not autonomous judgment."
- "The system keeps a human in the loop before any escalation is accepted."
- "The product direction is broader, but this demo is intentionally scoped to one operational pattern."

## open_questions
- Which exact statements from the master doc should be softened before live pitching?
- Does the team want to label the Forecaster as `predictive` or `prototype early warning`?
- Will judges be able to inspect source receipts directly in the UI, or only hear them narrated?

## provable_on_stage
- the UI flow from alert to recommendation to approval
- the existence of four reasoning stages in the console
- persona-specific rendering of the same alert
- the replayed evidence sequence for the selected incident

## defensible_if_asked
- the system can evolve into live adapters later because the data model separates replay from runtime sources
- heuristic contradiction checks can catch deception patterns such as impossible movement or ownership inconsistency
- the demo architecture is intentionally designed for degraded-network operation

## remove_from_pitch
- "predicts adversary behavior in real time" unless backed by visible logic and careful wording
- "fuses maritime, aviation, cyber, news, and satellite feeds live" unless those feeds are actually present
- "tells operators what is about to happen" as a blanket claim across all incidents

## credibility_repairs
- replace `prediction` with `early warning` where possible
- replace `agent swarm` with `four reasoning stages` unless runtime traces are real
- label replayed or prototype logic in the UI or demo script, not just in backup answers
